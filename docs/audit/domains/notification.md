# Аудит домена: Уведомления — in-app, диспетч, Telegram-бот, линковка

## 1. Назначение

Bounded-context `app/Domain/Notification` обслуживает два независимых механизма доставки оповещений сотрудникам CRM. Первый — **in-app уведомления** (спринт `feat/deals-redesign`): recipient-scoped таблица `notifications` питает колокольчик с flyout-поповером в плавающей навигации «Орбита» (actionable-корзина + хронологическая лента + дайджест по категориям), наполняется синхронными event-листенерами при назначении задачи и запросе согласования. Второй — **Telegram-бот** (спринт S2.9, на `nutgram`): привязка аккаунта через одноразовые deeplink-токены `/start link_<token>`, постинг карточки согласования в групповой чат с inline-кнопками голосования и отправка вердикта автору в личку.

**Зрелость: частично (бэкенд зрелый, end-to-end деградирован).** Оба бэкенда полностью разведены и соответствуют `ARCHITECTURE.md` (тонкие контроллеры, логика в сервисах, FormRequests, recipient-scoping через Policy, очереди для Telegram I/O, fail-closed редемпшн токена с `lockForUpdate`, нет секретов в `callback_data`, HTML-escaping). Живая схема (`notifications`, `telegram_link_tokens`) совпадает с миграциями 1:1, включая add-миграцию `action_label` и все индексы. Но **обе таблицы пусты** (`notifications=0`, `telegram_link_tokens=0`), и фича сломана с фронта двумя дефектами: (a) кнопка «Привязать Telegram» бьётся на контракте FE↔BE (FE читает `res.link_url`, API отдаёт `{deeplink, expires_in_minutes}`) — линковка из UI невозможна (вероятная реальная причина `telegram_link_tokens=0`); (b) весь in-app UI (колокольчик + flyout + бейдж) живёт **только внутри «Орбиты»**, которая не монтируется в дефолтном режиме навигации `sidebar` — из коробки пользователь колокольчика не видит (подтверждено в браузере, B.8). Итог: серверная часть качественная, но пользователь до неё не дотягивается.

## 3. Модель данных и реальность БД

| Модель | Таблица | Назначение | Строк в БД | Статус |
|---|---|---|---|---|
| `Notification` | `notifications` | Одно in-app уведомление одному получателю; источник истины для flyout (actionable + лента + дайджест). Поля: `id, user_id (FK users cascade), category (enum task/approval/deal/mention/system), title, body, is_actionable, action_label, deep_link, data (json), read_at, created_at` (без `updated_at` — immutable) | 0 | built |
| `TelegramLinkToken` | `telegram_link_tokens` | Одноразовый deeplink-токен «User↔Telegram». Валиден iff `used_at IS NULL AND expires_at>now()`; `used_at` штампуется (не удаляется) для аудита. Поля: `id, user_id (FK cascade), token (unique, $hidden), expires_at, used_at, timestamps` | 0 | built |
| `User.telegram_user_id` (колонка) | `users` | Привязанный Telegram chat_id/user_id после редемпшна; чистится при unlink. `varchar(255)`, индекс `users_telegram_user_id_index` | n/a | built |
| `Document.telegram_message_id` (колонка) | `documents` (owner=Contracts) | `message_id` карточки согласования для редактирования/снятия inline-клавиатуры после вердикта; идемпотентный resend | n/a | built |

**Расхождения migration ↔ live-schema ↔ model:**
- **Нет расхождений схемы.** `notifications` + `telegram_link_tokens` совпадают с миграциями 1:1 (колонки, типы, FK cascade, индексы `notifications_user_id_read_at_index`, `notifications_category_index`, `telegram_link_tokens_token_unique`, `telegram_link_tokens_expires_at_index`, `users_telegram_user_id_index`). `action_label` добавлен отдельной миграцией `2026_06_22`, присутствует в живой схеме.
- **Vault↔code drift (не migration↔live):** в vault-доке упоминается поле `action_url`, которого НЕТ — реальное поле модели/колонки `deep_link`. Vault-перечисление enum `(action | info)` НЕ соответствует реальному 5-значному `NotificationCategory`. Подробнее — §7.
- **Пустые таблицы при наличии кода:** обе таблицы пусты. In-app-сторона разведена (события диспатчатся из реальных `ActivityService`/`ApprovalService`) → 0 строк = непрогнанная тестовая БД. `telegram_link_tokens=0` дополнительно согласуется со сломанной FE-кнопкой линковки (токены создаются сервером и TTL-протухают неиспользованными).

## 5. RBAC домена

- **In-app лента (список/чтение/count/mark read single/batch/all):** любой авторизованный, **только свои** уведомления — без role/admin-override (by design). Двойная защита: `NotificationPolicy` (`view/update`: `user_id===caller`; `viewAny:true`) **и** жёсткий `WHERE user_id=caller` в каждом запросе/апдейте `NotificationService` (`api.php:415-423`, `NotificationPolicy.php:23-32`, `NotificationService.php:74,234-245`). Дыр в авторизации нет; две minor-заметки про ID-oracle (см. бэклог) — приемлемы для внутренней CRM.
- **Issue/unlink собственной Telegram-привязки:** любой авторизованный, только свою. `TelegramLinkController` действует исключительно на `request->user()`, cross-user-пути нет (`TelegramLinkController.php:28-40`, `api.php:408-409`).
- **Голос по согласованию через inline-кнопку:** только назначенные approver'ы стадии (привязанные TG). Сервер пере-проверяет membership стадии внутри `ApprovalService::decide` на каждом тапе; `ApprovalCallbackHandler` сначала резолвит user по `telegram_user_id` (`ApprovalCallbackHandler.php:43-57`, `ApprovalDecisionFlow.php:44`). Авторизация надёжная.

Главная RBAC-особенность: модель «только свои» реализована избыточно (Policy + сервисный scope), что хорошо. Единственный «недо-режим» — не авторизационный, а **UX-видимости**: в дефолтном `sidebar` колокольчика нет вовсе (major #2).

### MAJOR · BUG · ✅ подтверждено (static end-to-end trace; ранее live-probe вернул `{deeplink,...}`)
**Кнопка «Привязать Telegram» нерабочая: FE читает `res.link_url`, API отдаёт `{deeplink, expires_in_minutes}`**

- **Файлы:** `front/src/api/auth.ts:69-71` · `front/src/pages/ProfilePage/composables/useProfilePage.ts:179-180` · `src/app/Http/Resources/Notification/TelegramLinkResource.php:29-33` · `src/app/Domain/Notification/Services/TelegramLinkService.php:43-46`
- **Что происходит:** `authApi.telegramLink()` типизирован `Promise<{link_url:string}>`, а `linkTelegram()` вызывает `window.open(res.link_url,'_blank')`. Бэкенд `TelegramLinkResource::toArray` отдаёт `['deeplink'=>..., 'expires_in_minutes'=>...]`, питается `TelegramLinkService::issueFor` (`{deeplink, expires_in_minutes}`), ключа `link_url` НЕТ нигде. Верификатор прогрепал весь BE: `grep 'link_url'` по `src/**/*.php` = 0 совпадений (нет ремапа в контроллере/ресурсе/middleware); `grep 'deeplink'` по `front/src` = только doc-коммент (`auth.ts:67`), FE не читает `res.deeplink`. Ранее live POST `/api/me/telegram-link` под manager1 вернул `{"deeplink":"https://t.me/Contract_generator_MACRO_bot?start=link_TxMabDaB...","expires_in_minutes":10}`. Vault-док тоже фиксирует контракт `{deeplink, expires_in_minutes}` — то есть **FE неправ, а не API.** `res.link_url` структурно `undefined` → открывается пустая вкладка; 5с/60с poll (`useProfilePage.ts:156-174`) молча таймаутится без ошибки пользователю.
- **Repro:** Login → Profile → `?tab=telegram` → «Привязать Telegram». Открывается пустая вкладка (`window.open(undefined)`), info-toast + 60с poll стартуют, но не резолвятся. Линковка из UI невозможна. Токен создаётся на сервере, но осиротевший TTL-протухает — отсюда `telegram_link_tokens=0` при наличии кнопки.
- **Предлагаемый фикс:** В `front/src/api/auth.ts` сменить тип на `{deeplink:string; expires_in_minutes:number}` и читать `res.deeplink`; в `useProfilePage.linkTelegram` — `window.open(res.deeplink,'_blank')`. **НЕ** алиасить `deeplink→link_url` на бэкенде — vault + ресурс закоммичены на `deeplink`. Фикс чисто FE.

### MAJOR · MISSING · ✅ подтверждено (static) · 🌐 подтверждено в браузере (B.8)

### minor / trivial (не верифицировано — Phase-1)

- **minor · SPEC-DRIFT — Vault-док устарел vs код.** `Notification — Telegram-бот.md` описывает `NotificationCategory` как `(action|info)`, дайджест-чипы `action N / info N`, колонку `action_url`, листенеры `NotifyActivityAssignee`/`NotifyApprovers` на событии `ActivityCreated`, shape `{data, unread_count, counts_by_category}`. Реально: enum 5 значений `task/approval/deal/mention/system` (`NotificationCategory.php:14-18`); поле `deep_link` (не `action_url`); листенеры `NotifyActivityAssigneeListener`/`NotifyApproversListener` на `ActivityAssigned`; GET отдаёт `{actionable, feed:{data,meta}, digest:{unread_total,by_category}, unread_count}` (live). Код канонический, FE совпадает с кодом. Фикс — переписать vault (§7).
- **minor · BUG — Бейдж unread не обновляется после mount.** `fetchUnreadCount()` дёргается только в `NotificationsButton onMounted` (`NotificationsButton.vue:249-251`). Нет `setInterval`, нет websocket/Echo. Бейдж ресинкается лишь при open (`load → syncCount`) / close (`read-batch → syncCount`) flyout. Уведомление, созданное пока приложение idle, бейдж не бампит. Фикс: лёгкий interval-poll `/api/notifications/count` каждые 30–60с в store/кнопке, либо broadcast-канал; использовать дешёвый `/count`, не grouped.
- **minor · BUG — Привязанный Telegram показан как `@`+chat_id.** `telegramUsername = userStore.getUser?.telegram_user_id` (`useProfilePage.ts:137`), шаблон рендерит `«@{{ telegramUsername }}»` (`index.vue:232-234`). Но `telegram_user_id` хранит числовой chat/user id (`redeem` пишет туда chat id). Юзер видит `«@123456789»` — читается как handle, а это внутренний id. Фикс: убрать `@` и подписать как connected account/id, либо тянуть реальный `@username` с сервера. Косметика.
- **minor · SECURITY — Single mark-read даёт 403 (не 404) на чужое уведомление.** `POST /api/notifications/{notification}/read` через implicit binding + `authorize('update',...)` (`NotificationController.php:74-80`). Несуществующий id → 404, чужой реальный id → 403; разница позволяет авторизованному различать «id существует» vs «нет». Тело не утекает, id не чувствительны → impact низкий. Фикс (если нужна non-enumeration): scope binding на caller (`scopeBindings` / `where user_id`) чтобы чужие id давали 404.
- **minor · SECURITY — `read-batch` `exists:notifications,id` — глобальный ID oracle.** Правило без user-scope (`ReadBatchNotificationsRequest.php:38`, коммент явно «exists намеренно НЕ user-scoped»). Чужой существующий id проходит валидацию (нет 422), несуществующий — падает; валидатор различает существование независимо от владельца. Сервис молча пропускает чужие id (`WHERE user_id`), ничего не читается. Vault-бэклог явно принимает это как non-blocker. Фикс опционален: убрать `exists` (положиться на user-scoped `WHERE`) либо `Rule\Exists` с user-scope.
- **minor · SECURITY — Guard `telegram_user_id` scope только на `token.user_id`.** `redeem()` блокирует линковку только если `telegram_user_id` уже привязан к ДРУГОМУ юзеру (`TelegramLinkService.php:75-83`). НЕ блокирует, если у ТОГО ЖЕ юзера уже есть другой `telegram_user_id` — редемпшн свежего токена с другого TG-аккаунта (не привязанного к иному) тихо перезатирает (`75-92`). Токен owner-issued, актор — легитимный владелец → impact низкий (смена устройства), но без confirmation и прежний TG молча осиротеет. Фикс (опц.): при наличии другого `telegram_user_id` требовать явный unlink или отдавать `already_linked_overwrite` для подтверждения ботом. (confidence 0.6)
- **trivial · DATA-INCONSISTENCY — `digest.by_category` как `[]` вместо `{}` при пустом.** `NotificationService::digest` строит `$byCategory=[]`; при нуле непрочитанных цикл не выполняется → `json_encode` отдаёт `"by_category":[]` (live: `digest:{"unread_total":0,"by_category":[]}`). FE типизирован `Record<string,number>`, читает через `Object.keys(... ?? {})` — терпит пустой массив, flyout работает, но контракт array-vs-object несогласован. Фикс: `(object) $byCategory` в сервисе или обернуть `(object)` в `NotificationFeedResource`. Чисто BE.
- **trivial · CONVENTION — FE «mark all read» отбрасывает server-count.** `POST /api/notifications/read-all` отдаёт `{marked, unread_count}` (`NotificationController.php:84-92`), но `notificationsApi.markAllRead()` типизирован `Promise<void>` и игнорит тело; композабл оптимистично `clearCount()`. Single-read и read-batch используют `syncCount` — это несогласованность, не утечка (mark-all и так обнуляет). Фикс: типизировать как `Promise<{marked,unread_count}>` и `syncCount(res.unread_count)`.
