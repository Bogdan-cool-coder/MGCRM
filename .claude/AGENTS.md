# AGENTS.md — Governance MACRO Global CRM Agent Fleet

> **Активный стандарт организации агентного флота MGCRM.** Этот документ кодифицирует модель организации флота, согласованную в стратегической дискуссии, и действует вместе с `CLAUDE.md` + `.claude/agents/`.

---

## 0. Зачем этот документ

Текущие 17 агентов выросли стихийно: горизонтальные и доменные роли смешаны, нет явных правил масштабирования, нет governance для split/spawn. Это рождает координационную сложность O(N²) при полной mesh-сетке, когда каждый доменный агент дрейфует от каждого другого по общим требованиям (стек, DSL, тесты, токены).

Этот документ фиксирует **гибридно-матричную модель** с двухуровневой иерархией (hub-and-spoke), разделением горизонтальных владельцев и доменных исполнителей, и правилами масштабирования, пропорциональными задаче.

**Обоснование гибрида над полными силосами:** hub из 5 горизонтальных владельцев даёт O(N) при добавлении новых модулей (доменные агенты подключаются к константному хабу); полная mesh N×N агентов даёт O(N²) дрейфа по общим требованиям. Это главный архитектурный довод в пользу матрицы.

**Многоагентность обоснована только для задач с:**
- параллелизуемой работой (backend + frontend независимы);
- контекстом, превышающим одно окно;
- большим числом инструментов по разным зонам.

Она НЕ обоснована для тесно связанного кода с общими зависимостями: multi-agent потребляет ~15× токенов при синхронной работе. **Правило: задача → один агент, пока не доказано обратное.**

## 1. Иерархия: два уровня, не больше

```
main (оркестратор)
├── горизонтальные владельцы (хаб, 5 агентов — константа)
│   ├── reviewer         ← переименован из product-manager
│   ├── backend-architect  ← переименован из backend-specialist
│   ├── designer
│   ├── qa-tester
│   └── deploy-engineer
└── доменные исполнители (споуки, по модулю)
    ├── <module>-backender
    └── <module>-frontender
```

**Жёсткое правило двух уровней:** субагенты НЕ порождают собственных субагентов. Только `main` спаунит и секвенирует агентов. Субагент выполняет работу и возвращает результат main — не цепочку.

**main не читает `src/` для ревью.** Это нарушает высоту обзора и размывает роль. Ревью — задача `reviewer`.

## 2. Горизонтальные владельцы (хаб, 5 ролей)

Хаб — константного размера. Новые модули не добавляют новых горизонтальных ролей.

### 2.1 `reviewer` (бывший `product-manager`)

**Роль:** финальное звено перед апрувом после каждой итерации рабочего агента. Глубокий code review + проверка паритета old→new + doc-sync.

**Почему reviewer, а не PM:** задача требует глубокого чтения кода и свежего независимого контекста. Складывание её в `main` нарушает правило «main не читает src». Тонкий `reviewer` сохраняет независимую проверку.

**Частичное поглощение Product main'ом:** роутинг задач + фреймирование (цель + границы + ожидаемый выход) + принятие решения о готовности (из distilled summary reviewer'а) + steering консистентности (флагует cross-module overlap при диспетчеризации) — это делает main. Reviewer удерживает code-depth. **Они не дублируются — они дополняют.**

**Инструменты:** Read, Edit, Write, Grep, Glob, Bash (только git + diff). Не пишет код.

### 2.2 `backend-architect` (бывший `backend-specialist`)

**Роль:** хранитель backend-консистентности всего проекта.

**Rescope (что изменилось vs backend-specialist):**
- Прежде: исполнитель инфраструктурного кода (auth, миграции, тесты).
- Теперь: **spec-author + review-gate** для backend-границ доменов. Пишет и ownit инта-модульный API-контракт (форма Service, сигнатуры, события). Ревьюит cross-domain зависимости. Предотвращает дублирование логики между доменными агентами. Ведёт реестр библиотек (library-registry): если задачу закрывает существующий пакет — не ставим новый.

**Что backend-architect НЕ делает:** не управляет субагентами в реальном времени; не пишет доменную логику сам — пишет контракт, по которому её пишет `<module>-backender`.

**Ключевой артефакт:** `backend-architect` ownit backend-эталон (подлежит написанию: **`docs/backend-standard.md`**) — конкретный house-style с worked-примерами: таблица доменных границ, worked cross-slice пример, reuse-чеклист, library-registry. Это эталон стека вместе с `ARCHITECTURE.md`.

**Последовательность в workflow:** main секвенирует backender ПЕРЕД frontender — контракт (endpoint, response shape) должен существовать до того, как frontender начинает строить.

### 2.3 `designer`

**Роль:** owner дизайн-системы и frontend-консистентности. Авторит ТЗ на уровне компонентов перед любой UI-задачей. Сторожит визуальную согласованность между модулями.

**Source-of-truth цепочка дизайна:**
1. Значения токенов → `front/src/theme` (код, единственный источник).
2. Визуальный замысел конкретного экрана → `design-handoff/redesign/` (спеки + мокапы).
3. Бренд-инварианты / общая дизайн-система → skill `macroglobal-design` (`.claude/skills/macroglobal-design/`).

**Компонентный инвентарь:** designer ведёт лёгкий документ-инвентарь компонентов (reuse-first перед написанием нового). Storybook отложен до момента, когда shared-library оправдает инфраструктуру.

**21st.dev/magic MCP:** НЕ используем как код (эмитирует React+Tailwind — чужой стек). Максимум — свободное визуальное вдохновение.

### 2.4 `qa-tester`

**Роль:** целопродуктовый QA-гейт после любого UI-изменения. Единственный агент с доступом к браузерным MCP (Claude_in_Chrome / chrome-devtools / Control_Chrome / Claude_Preview). Визуальный гейт design-system: computed-styles в обеих темах; отклонение = FAIL.

**Когда НЕ нужен:** backend-only, рефакторинг без UI, правки только `.md`/`.yml`/`.json`, правки агентов/CLAUDE/PLAN/ARCHITECTURE.

**Playwright/chrome-devtools limitation:** нельзя коммитить выборы PrimeVue overlay через automation. Для dropdown-dependent writes — нативный click + input events или API.

### 2.5 `deploy-engineer`

**Роль:** владелец деплой-инфры (Docker/GHA/SSH). Единственный агент, делающий git push в `main`.

**Жёстко:** push в `main` — **ТОЛЬКО по явной прямой просьбе юзера**. Push автоматически триггерит прод через `deploy.yml`. WIP в `main` не пушим. Изменения деплой-инфры — только по явной просьбе.

## 3. Доменные исполнители (споуки)

### 3.1 Модель split по слоям

Каждый активный модуль получает **два** исполнителя вместо одного `<domain>-specialist`:

| Агент | Зона | Инструменты |
|---|---|---|
| `<module>-backender` | Domain сервисы, миграции, тесты, API Resource, Policy | Read, Edit, Write, Bash, Grep, Glob |
| `<module>-frontender` | Vue-страницы, composables, Pinia, SCSS этого модуля | Read, Edit, Write, Bash, Grep, Glob |

**Причина split:** backend и frontend — параллелизуемые задачи с разным контекстом. Один агент с обеими зонами тратит контекстное окно на несвязанный код; два специализированных читают только своё.

**Последовательность:** main запускает backender ПЕРВЫМ → backender фиксирует endpoint + response shape → main запускает frontender с этим контрактом. Frontender не строит на «предполагаемом» API.

### 3.2 Правила масштабирования (task-proportional)

**Не создавай всех агентов заранее.** Split обоснован только для активных/тяжёлых модулей.

| Размер задачи | Кто делает |
|---|---|
| Тривиальная правка (1-2 файла, нет UI) | main или 1 specialist |
| Средняя фича (backend + frontend, изолированный модуль) | 2-4 агентов |
| Большой аудит / multi-domain / redesign | полная команда |

**Pilot сначала, масштаб потом:** split по `sales` и `crm` (самые тяжёлые модули) — первыми. Остальные модули остаются на `<domain>-specialist` до момента, пока задача не оправдает split.

**Spawn-on-demand:** новый `<module>-backender` / `<module>-frontender` создаётся только когда есть реальная задача для этого модуля, не заранее.

### 3.3 Текущий реестр доменных агентов

Статус на момент написания документа (реальный, не M-номера):

| Агент (текущий slug) | Модуль | Статус | Путь к split |
|---|---|---|---|
| `crm-specialist` | Crm | построено | pilot-кандидат #1 |
| `sales-specialist` | Sales + SalesPulse | построено (LIVE) | pilot-кандидат #2 |
| `contract-specialist` | Contracts | каркас, не в проде | при следующем спринте |
| `onboarding-specialist` | Onboarding | backend построен, student-loop сломан | при след. спринте |
| `cs-specialist` | CustomerSuccess | greenfield | создать при старте спринта CS |
| `finance-specialist` | Finance | greenfield | создать при старте спринта Финансы |
| `automation-specialist` | Automation | каркас, движок не запускался в проде | при след. спринте |
| `integration-specialist` | Integration (вшит в Inbox/Notification) | нет standalone Domain | при старте спринта |
| `analytics-specialist` | Analytics + Log (вшит в Sales) | нет standalone Domain | при старте спринта |
| `bot-specialist` | Bot | — | при старте спринта |
| `migration-specialist` | Migration (cutover) | только на закрывающем спринте | не сплитить |

**Переходный период:** текущие `<domain>-specialist` агенты продолжают работу, пока модуль не сплитнут. Rename/split конкретного модуля — только по явному решению юзера.

## 4. Роль main (оркестратор)

### 4.1 Что делает main (whitelist)

1. Диалог с юзером (AskUserQuestion, summary).
2. Оркестрация tasks — секвенирование, не исполнение.
3. Делегирование (Agent) с явным фреймингом: цель + границы + ожидаемый выход.
4. Steering консистентности при диспетчеризации: если задача затрагивает два модуля — подтягивает нужных горизонтальных владельцев.
5. Локальный git: `status/diff/log/add/commit` (без push) после reviewer-аппрува.
6. Запись секретов в `src/.env`.
7. Чтение memory/CLAUDE.md/PLAN.md/ARCHITECTURE.md для ориентации.
8. Минимальный Read/Grep для маршрутизации (НЕ для починки бага).

### 4.2 Анти-паттерны main (запрещено)

- Read src-файла «чтобы понять класс».
- grep ради поиска бага.
- `docker compose exec … artisan …` напрямую.
- SSH на сервер.
- Правки кода/тестов/.md «по мелочи» самостоятельно.
- Ревью diff — это reviewer.

### 4.3 Фрейминг задачи при делегировании

При спауне агента main явно передаёт:
- **Цель:** что нужно получить (поведение, не технологию).
- **Границы:** какие файлы/домены трогать, каких избегать.
- **Ожидаемый выход:** что агент возвращает (distilled summary, конкретные файлы, тест-результат).
- **Контракт** (для frontender): endpoint + response shape от backender-а, если они есть.

## 5. Гарантии изоляции и граничные контракты

### 5.1 Worktree-изоляция

По умолчанию каждый агент работает в отдельном git-worktree — не в общем working tree. Инцидент со stash/reset при общем дереве показал риск: один агент сбрасывает незакоммиченную работу другого.

**Правило:** при параллельном запуске backender + frontender — раздельные worktree'и, объединяются через merge/rebase перед commit.

### 5.2 Модульные границы (DDD)

Cross-domain доступ — **только через Service публичного API доменного владельца**, никогда прямым Eloquent в чужую модель. Это закон ARCHITECTURE.md, гарантируемый `backend-architect` на ревью.

```
Domain A Service ──► Domain B (только через B's Service)
Domain A Model   ✗── Domain B Model (запрещено напрямую)
```

### 5.3 Shared memory для конвенций

Конвенции (токены, стек-правила, naming, RBAC-состояние), используемые несколькими агентами, живут в:
- `CLAUDE.md` (always-injected, легковесный).
- `ARCHITECTURE.md` (детальные паттерны).
- `.claude/agents/<name>.md` (зона агента).
- `AGENTS.md` (этот документ, governance).

Агент НЕ дублирует конвенции в своём md — ссылается на источник. Избыточные копии расходятся и гниют.

## 7. Масштабирование флота (правила роста)

### 7.1 Когда НЕ делать multi-agent

- Задача тесно связана: общий контекст + взаимные зависимости → один агент быстрее и дешевле.
- До ~4-5 активных модулей параллельно → hub достаточен без group-lead.

### 7.2 Когда сплитить модуль

Split `<module>-specialist` → `<module>-backender` + `<module>-frontender` обоснован если:
- Модуль активно развивается (2+ спринта подряд).
- Backend и frontend задачи в спринте параллелизуемы (не ждут друг друга по контексту).
- Один агент регулярно упирается в лимит контекста.

Решение о split принимает юзер, не main и не агент.

### 7.3 Cluster-lead при большом флоте (>15-20 модулей)

При росте флота сверх 15-20 модулей — optional group-lead для кластера (~3-4 модуля), фрактальный гибрид. До этого порога — хаб из 5 горизонтальных владельцев + прямые споуки.

**Сейчас** (10 доменных модулей) — cluster-lead не нужен, хаб справляется.

### 7.4 Machine-enforced конвенции

По мере роста флота вместо ручного ревью — автоматические гейты:
- `npm run lint:ds` (stylelint, pre-commit + CI) — дизайн-система.
- `php artisan test` на SQLite :memory: (CI) — backend.
- Scaffold-генераторы (artisan stub) — единый шаблон нового домена.

Цель: O(1) распространение конвенций через tooling, не через повторение в каждом md.

## 9. Железные правила (общие для всего флота)

Эти правила дублируются здесь как reminder; канонический источник — `CLAUDE.md` + `ARCHITECTURE.md`.

- **ARCHITECTURE.md — закон.** Весь код строго по нему. Отклонение = баг; reviewer режет на ревью.
- **Стек жёсткий** (PLAN §3). Новый пакет — только по явной просьбе юзера + аппрув.
- **Commit — только English.** НИКАКИХ `Co-Authored-By: Claude`, упоминаний Claude/Anthropic/AI. Без `--no-verify`, `--force`.
- **Деструктив** (`down -v`, `DROP`, `rm -rf` данных) — только по явной просьбе + бэкап. Guard-хук блокирует.
- **Тесты — PHPUnit + SQLite :memory:** с тройной изоляцией. Тесты никогда не ходят в живую БД.
- **Push в `main` — только `deploy-engineer` по явной прямой просьбе юзера.** Push = прод (GHA `deploy.yml`). WIP в `main` не пушим.
- **Субагенты не порождают субагентов.** Только main спаунит агентов.
- **main не читает `src/` для ревью.** Это задача reviewer.

## 10. Глоссарий

| Термин | Определение |
|---|---|
| **hub** | 5 горизонтальных владельцев (reviewer, backend-architect, designer, qa-tester, deploy-engineer) |
| **spoke** | Доменный исполнитель (`<module>-backender` / `<module>-frontender`) |
| **distilled summary** | Краткий структурированный отчёт агента → main после завершения задачи |
| **intra-module API contract** | Форма Service + сигнатуры + endpoint, фиксируемые backender перед стартом frontender |
| **worktree** | Отдельная рабочая директория git для изолированной работы агента |
| **split** | Разделение `<domain>-specialist` на `<module>-backender` + `<module>-frontender` |
| **spawn-on-demand** | Агент создаётся только при наличии реальной задачи, не заранее |
| **IAM-1** | ЗАКРЫТ: авторизация переведена на spatie/laravel-permission + guard sanctum; `users.role` удалён (`role` — виртуальный accessor). Двойного источника роли больше нет |
